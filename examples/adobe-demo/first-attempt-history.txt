The issue is that `Asset ID` is lost after the `STATS` aggregation because you're only grouping by `Campaign Name`. After the STATS command, only the fields in the `BY` clause and the aggregated fields are available.

Here's the corrected query:

```esql
FROM campaign-performance
| EVAL roi = (Revenue - Spend) / Spend * 100
| STATS 
    total_spend = SUM(Spend),
    total_revenue = SUM(Revenue),
    avg_roi = AVG(roi),
    total_conversions = SUM(Conversions)
  BY `Campaign Name`
| EVAL roi_percentage = (total_revenue - total_spend) / total_spend * 100
| SORT roi_percentage DESC
| LIMIT 10
```

## Why This Happened

The original query tried to join with `brand-assets` **after** aggregating by campaign name. At that point:
- You have one row per campaign (not per asset)
- `Asset ID` no longer exists in the result set
- The join can't work because there's no `Asset ID` to join on

## Alternative: If You Want Asset Enrichment

If you actually want to enrich with asset information, you need to join **before** the aggregation:

```esql
FROM campaign-performance
| LOOKUP JOIN brand-assets ON `Asset ID`
| EVAL roi = (Revenue - Spend) / Spend * 100
| STATS 
    total_spend = SUM(Spend),
    total_revenue = SUM(Revenue),
    avg_roi = AVG(roi),
    total_conversions = SUM(Conversions)
  BY `Campaign Name`, `Asset Type`
| EVAL roi_percentage = (total_revenue - total_spend) / total_spend * 100
| SORT roi_percentage DESC
| LIMIT 10
```

This version:
1. Joins first (so you have asset type available)
2. Groups by both campaign AND asset type
3. Shows ROI broken down by campaign and asset type

| EVAL ctr = Clicks/Impressions

returns values of 0 for all ctr and cvr fields, despite Clicks, Conversions, and Impressions having values

The issue is that you're doing **integer division** in ES|QL! When you divide two integers, ES|QL returns an integer result, and since Clicks/Impressions is always less than 1, it truncates to 0.

## Fix: Force Floating Point Division

You need to cast at least one operand to a double to get decimal results:

```esql
FROM campaign-performance
| WHERE Date >= "2024-06-01" AND Date <= "2025-06-30"
| EVAL ctr = TO_DOUBLE(Clicks) / Impressions * 100
| EVAL cvr = TO_DOUBLE(Conversions) / Clicks * 100
| STATS 
    total_impressions = SUM(Impressions),
    avg_ctr = AVG(ctr),
    avg_cvr = AVG(cvr),
    total_revenue = SUM(Revenue)
  BY Region, Channel
| WHERE total_impressions > 100000
| SORT total_revenue DESC
```

## Why This Works:

- `TO_DOUBLE(Clicks)` converts the integer to a floating-point number
- Once one operand is double, ES|QL performs floating-point division
- Multiplying by 100 gives you the percentage

## Alternative Approach:

You can also multiply first, then divide:

```esql
| EVAL ctr = Clicks * 100.0 / Impressions
| EVAL cvr = Conversions * 100.0 / Clicks
```

The `.0` after 100 forces it to be treated as a double.

## Full Corrected Query:

```esql
FROM campaign-performance
| WHERE Date >= "2024-06-01" AND Date <= "2025-06-30"
| EVAL ctr = TO_DOUBLE(Clicks) / Impressions * 100
| EVAL cvr = TO_DOUBLE(Conversions) / Clicks * 100
| STATS 
    total_impressions = SUM(Impressions),
    avg_ctr = AVG(ctr),
    avg_cvr = AVG(cvr),
    total_revenue = SUM(Revenue)
  BY Region, Channel
| WHERE total_impressions > 100000
| SORT total_revenue DESC
```
